# Database & Persistence Guidelines

> D1 (Drizzle ORM) + R2 + KV patterns, as built for SeatView (seatmap-real).
> Captured 2026-05-24 from the MVP implementation.

---

## Schema (D1 via Drizzle)

Source of truth: `src/server/db/schema.ts`. Migrations live in `migrations/`
(generated by `drizzle-kit`; apply with `wrangler d1 migrations apply`).

Only user-generated content lives in D1. Venue metadata is static JSON in
`data/venues/*.json` and bundled at build time (ADR-1) — never queried from D1.

### `photos`
`id` (text ulid, PK), `venue_id`, `sub_map_id`, `x_percent`/`y_percent` (real
0..1), `image_key` (R2 key), `seat_label` (required), `performance_date` /
`event_name` / `description` (nullable), **`width` / `height` (integer NOT NULL
— real intrinsic pixels of the uploaded WebP)**, `ip_hash` (salted SHA-256,
never raw IP), `created_at` (epoch ms), `deleted_at` (nullable, soft delete).
Index `idx_photos_venue (venue_id, sub_map_id, deleted_at)`.

> **Why width/height are stored** (migration `0001`): the masonry grid
> (`react-photo-album`) needs real aspect ratios up front to lay out without
> cropping or CLS. The locked product rule is "照片不裁切 / authenticity over
> neatness" (shape-photo-grid §10). Do NOT reintroduce a derived/pseudo aspect
> ratio — render at the stored ratio.

### `staging_venues`
`id` (ulid PK), `name` (verbatim user text, R9.5 — not normalized/translated),
`ip_hash`, `created_at`, `processed_at` (nullable; non-null ⇒ "已收录").

### Convention: soft delete (ADR-6)
Public reads MUST filter `deleted_at IS NULL` (`src/server/photos.ts`). Admin
delete sets `deleted_at` AND physically deletes the R2 object (`BUCKET.delete`).
R2 purge is best-effort: a purge failure does not roll back the soft delete.

## Scenario: photo upload write path (infra + cross-layer + DB)

### 1. Scope / Trigger
Anonymous client uploads a seat-view photo. Touches Turnstile, KV rate-limit,
R2 object storage, and a D1 insert — so the write MUST NOT be forgeable by the
client. This is the canonical anti-forgery pattern for this project.

> **Accepted deviation from ADR-12**: ADR-12 specified client-direct-to-R2 via
> a presigned URL. As built we use **bound-bucket direct write through the
> Worker** + an **HMAC ticket**, NOT presigned URLs. Reason: presigned URLs
> need R2 S3 credentials + CORS that are awkward locally, and image bytes are
> ≤500KB (client-compressed) so routing through the Worker is well within
> limits. Do not "restore" presigned URLs without a deliberate decision.

### 2. Signatures
- `POST /api/upload/sign`  → `{ ticket: string, ... }`  (`src/pages/api/upload/sign.ts`)
- `POST /api/upload/commit` (multipart: image + ticket) → `201 { photo: PhotoDto }` (`src/pages/api/upload/commit.ts`)
- `signTicket(payload) / verifyTicket(token)` (`src/server/upload-ticket.ts`)

### 3. Contracts
- **Sign request** (`SignRequest`, `src/lib/upload.ts`): `turnstileToken`,
  `venueId`, `subMapId`, `xPercent`, `yPercent`, `seatLabel`, optional
  `performanceDate`/`eventName`/`description`, **`width`, `height`** (ints,
  validated `1..IMAGE_MAX_EDGE=4096`).
- **Ticket payload** (HMAC-signed, `TicketPayload`): all upload fields INCLUDING
  `width`/`height` + an expiry. The signature makes every field tamper-proof.
- **Commit**: reads field values from the **verified ticket**, never from the
  multipart body (the body only carries the image bytes + the ticket string).
- **Env**: a single server secret `TURNSTILE_SECRET_KEY`, reused for three
  purposes — ① Turnstile siteverify, ② the upload HMAC ticket signing key
  (`src/server/upload-ticket.ts`), ③ the IP-hash salt (`src/server/ip.ts`).
  There is intentionally NO separate `UPLOAD_TICKET_SECRET` / `IP_HASH_SALT`
  (both files note this at their top). Plus bindings `DB` (D1), `BUCKET` (R2),
  `RATE_LIMIT` (KV) (+ the adapter-required `SESSION` KV). See `.dev.vars.example`.

### 4. Validation & Error Matrix
- Turnstile token missing/invalid → `400` (sign rejects before signing).
- IP over daily cap (10/day) or within 30s cooldown → `429` (`rate_limited_daily` / `rate_limited_cooldown`). Quota is consumed only AFTER a successful D1 write, so a failed Turnstile/insert does not burn quota.
- `width`/`height` not integer in `1..4096` → `400 missing_fields` (sign).
- Tampered/expired ticket at commit → `400 invalid_ticket` (4xx ⇒ no client retry).
- Network/5xx at commit → client retries (ADR-12: ≤2 retries, 1s/3s backoff, same Turnstile token + ticket reused).

### 5. Good/Base/Bad Cases
- Good: valid Turnstile + within limits → sign 200 → upload bytes + ticket → commit 201, row visible newest-first.
- Base: optional metadata omitted → row inserted with NULLs, grid/Lightbox render fine.
- Bad: client edits `seatLabel` in the commit body → ignored; the ticket value wins (un-forgeable).

### 6. Tests Required
- Unit: `verifyTicket` rejects tampered/expired tokens (assert `invalid_ticket`).
- Integration (local D1/KV/R2): 10 uploads pass, 11th → 429; cooldown blocks a 2nd within 30s; committed row has 64-hex `ip_hash`, no consent column (R11.4), stored `width`/`height` match the signed values.

### 7. Wrong vs Correct
- **Wrong**: `INSERT ... VALUES (body.seatLabel, body.width, ...)` — client controls the row.
- **Correct**: `const t = verifyTicket(form.ticket); INSERT ... VALUES (t.seatLabel, t.width, ...)`.

## ID generation (gotcha)

> **Warning**: the `ulid` npm package throws `"secure crypto unusable"` inside
> workerd. Use the project's `src/server/id.ts` (`crypto.getRandomValues`-based
> ULID) for all PKs. Do not `import { ulid } from "ulid"`.

## Rate-limit KV (shared util)

`src/server/rate-limit.ts` is generic over a `RateScope`. Reuse it; do not write
ad-hoc counters. Keys carry a TTL so they auto-expire:
`ratelimit:<scope>:<ip_hash>:<YYYY-MM-DD>` (daily) and
`ratelimit:upload:cooldown:<ip_hash>` (30s). Upload = 10/day + 30s cooldown;
staging = 5/day. `incrementDaily` is called only after the protected write
succeeds.
