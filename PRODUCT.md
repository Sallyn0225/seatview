# Product

## Register

product

## Users

SeatView 服务两类核心用户，且必须双语等价对待：

- **浏览者**：日本/海外演唱会粉丝，在抢票或选座前打开 SeatView，想知道"我准备买的这个座位，实际能看到什么"。语言为简体中文或日本语；进入网站的设备以移动端居多（地铁、休息时间、抢票排队时），桌面端为辅。**核心使用情境**：抢票窗口前几分钟、确认选座时的紧张时刻 —— 任何加载延迟或界面噪音都直接转化为决策焦虑。
- **贡献者**：去 Live 现场的粉丝，回家后想把自己座位的视角分享给社群参考。无注册系统，没有"我的主页"作为激励，**唯一驱动力是"帮其他粉丝避免踩雷"的利他感**。设备以桌面端为主（图片传输 / 表单填写）。

两类用户都不持续访问，多数是一次性、目的明确的访问。SeatView 不需要"留存"，需要的是"该用户当次找到答案，并下次再有同类问题时还想得起来回来"。

## Product Purpose

SeatView 聚合日本（含部分海外）演唱会场馆的**真实座位视角图**。用户在场馆官方坐席图上点击任一标注点，就能看到那个座位由其他粉丝实拍的现场视角；想贡献的人则在同一张坐席图上标注自己的位置并上传一张照片。

存在的理由是一个具体的市场缺口：日本主流票务平台（ticket.pia、e-plus、各场馆官网）只给出抽象的坐席图，从未告诉用户"E 列 23 号的实际视野"。一手信息散落在 X (Twitter)、个人博客、Live fans BBS 里，难以按场馆/区域聚合检索。SeatView 把这件事做成**结构化、可索引、按场馆分类、按座位定位**的图集。

成功的样子：粉丝在"选座犹豫的 30 秒"内能找到至少 1 张参考图，并因此减少一次糟糕的购票决策。**不是**追求 MAU、用户停留时长、UGC 数量绝对值 —— 这些都是反指标。

## Brand Personality

**日式克制现代 / 双语雅气 / 资料图鉴感**。三个关键词：

- **克制（克）**：界面让位于内容。照片和坐席图占据视觉绝对主导，chrome（导航 / 操作按钮 / 装饰）必须能在用户注视图片时"消失"。
- **真实（真）**：照片是用户用手机拍的，可能有反光、人头、模糊 —— 这种粗糙感本身就是 SeatView 区别于票务平台精修宣传图的核心价值。设计不能去"美化"它，而要把它呈现得诚实、清晰、值得信任。
- **雅气（雅）**：保留日式排版的呼吸感 —— 大量留白、细线分隔、必要时让明朝/衬线字体点缀标题或品牌符号。**但不能滑向极简冷漠**：空状态、上传成功、初次访问等"缝隙时刻"允许小幅度的人情味（一句温柔的文案、一个克制的图形点缀），主流 UI 仍然严肃。

声音 / 文案：第二人称、平实、不卖弄、不感叹。日文与中文等价，**不分主次、不带翻译腔**。不用「请」「敬请」这类过度礼貌堆叠，也不用「亲」「小可爱」这类社交平台式套近乎。

参考：[muji.com](https://www.muji.com/jp/)、[cinra.net](https://www.cinra.net/)、[cookpad UED](https://ued.cookpad.com/)、[Are.na](https://www.are.na/)、早期 Pinterest。

## Anti-references

明确要避开的三类视觉/交互模板：

1. **日本票务 & Live 资讯站的拥挤老旧**：[eventernote](https://www.eventernote.com/)、[livefans.jp](https://www.livefans.jp/)、[ticket.pia](https://t.pia.jp/)、[e-plus](https://eplus.jp/) —— 90 年代 BBS 排版、滚动公告条、大量小字密集堆叠、广告位与内容混排。SeatView 哪怕场馆树有 47 个分区，也必须看起来呼吸顺畅、可一眼定位。
2. **标准 SaaS 渐变模板**：紫蓝渐变 hero、big-number + small-label 卡片网格、icon + heading + 三行说明的卡片重复、glassmorphism、彩色渐变 CTA。SeatView 是一个图鉴工具，不是 SaaS 产品演示页 —— 任何"hero metric"式的展示都不存在。
3. **极简到冷漠**：纯白底 + 纯黑字 + 无任何温度的 chrome、看起来像未完成的开发者 demo。克制 ≠ 没人住过。即使最严肃的列表页也要在排版节奏、字体细节、轻微的颜色选择上透出"这是个被人精心维护的小项目"的感觉。

另外两条隐性反例（来自 prd.md 的产品决策）：

- **不要社交平台式喧闹**：没有点赞数、follow 按钮、彩色徽章、推送红点 —— SeatView 无注册系统、不做评论评分（prd "Out of Scope"），UI 也不能假装有这些功能。
- **不要"模态优先"的交互**：除了座位视角的 Lightbox 是必须的（图片预览天然需要全屏聚焦），其他操作能 inline 就 inline，能 sheet 就 sheet。上传流程是 sheet/drawer，不是层层弹窗。

## Design Principles

1. **图片是主角，UI 是背景**：用户上传的视角照片必须占据每个详情视图至少 70% 的视觉重量。任何 chrome（标题、按钮、面包屑、tab）都必须能在用户聚焦图片时"褪到背景"。这是 SeatView 与票务平台 / 资讯站的根本差异 —— 后者把图片缩成缩略图、用文字和广告填满版面。
2. **克而不冷，温度藏在缝隙里**：主体 UI（场馆树、搜索、瀑布流、坐席图）保持严肃克制；但**空状态、错误提示、初次访问的说明页、上传成功的反馈** —— 这些"缝隙时刻"允许出现一句温柔的文案、一个克制的图形点缀。判断准则：用户在完成任务途中看到的东西必须不打扰，用户在"暂停的瞬间"看到的东西可以有人味儿。
3. **双语等价，不偏废任何一种**：中文和日文是 SeatView 的两条核心轨道，**不是"主语言 + 翻译"**。排版上互不挤压（不强行中日并列，但切换时整个页面气质必须保持一致）、字号节奏一致、文案各自地道（不用机翻腔，不用"中文写好再翻日文"的语序）。场馆名是中日双语字段并列保存，用户切换语言时连场馆名都同步切换。
4. **无注册即贡献，信任靠透明而非账户**：SeatView 不要求用户注册就能上传 —— 这是产品定义。但这意味着信任不能依赖"账户体系 + 用户身份"，必须依赖**流程透明度**：上传时为什么要勾版权同意框（CC BY-NC 4.0）、为什么要做 Turnstile、为什么 IP 限频 10 张/天 —— 每一步都要在界面上**用一句话讲清楚原因**，不让用户感觉"被默默监控"。
5. **真实图像第一，元数据次之**：座位号、演出日期、活动名、描述都是辅助字段。它们的视觉呈现绝不能与图片争重量 —— 字号、颜色、间距都要明显低于图片本身。在 Lightbox 里这一点尤其关键：照片是焦点，文字是脚注。
6. **抢票时刻的零摩擦**：浏览路径（场馆树 → 场馆页 → 坐席图 → 点击标注 → Lightbox）必须**毫秒级响应**、移动端单手可达、不出现任何"先看广告"或"先选语言"的拦截。说明页只在第一次访问时出现，之后 localStorage 让用户直接回到上次的场馆。这一条是其他所有原则的底线 —— 即使界面再美，慢一秒、多点一次都是失败。

## Accessibility & Inclusion

**基线：WCAG 2.1 AA**。所有 craft / polish / audit 工作以此为只需达到的门槛，不强制 AAA。

具体要求：

- 文字与背景对比度 ≥ 4.5:1（小字）、3:1（大字 18pt+）。OKLCH 调色时主动检查。
- 完整键盘可达：所有交互（场馆树折叠/展开、搜索、坐席图标注点点击、Lightbox 打开/关闭、上传流程）能用 Tab/Enter/Esc 操作完成。
- `prefers-reduced-motion` 必须尊重：聚合点放大 + 散开动画（ADR-4）在 reduced motion 下降级为即时跳转。
- 所有图片都有 `alt` 文本：场馆坐席图用场馆名 + sub-map 标签；用户上传的视角图用 `seat_label` + 演出日期（若有）。
- 双指缩放/平移在移动端必须正常工作（R3.8）—— 不能为了"防止误触"禁用浏览器默认缩放。
- 颜色不是唯一的信息通道：标注点的状态（已聚合 / 已选中 / 默认）必须同时通过形状或大小区分，不能仅靠颜色。
- 双语切换不依赖鼠标精确点击：右上角语言切换器必须有 ≥ 44×44px 的点击区域。

不强制 AAA 的原因：SeatView 主要服务粉丝群体而非公共服务，提高到 AAA 会限制色彩表现力（特别是日式克制风格依赖的低对比微差），收益不抵成本。但若未来接入到任何公共售票渠道（如官方合作），应重新评估升级到 AAA。
